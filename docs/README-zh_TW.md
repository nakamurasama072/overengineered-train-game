# overengineered-train-game

## 一個將 Java 設計模式硬搬進 Python 的警示案例

這個儲存庫包含《Ticket to Ride》桌遊的 Python 實作。但它的目的**不是**作為一個可運作的正確專案，而是一個**教育警示**：告訴你將 Java 的設計模式「生搬硬套」到 Python 會有多可怕。

程式碼極度過度設計、難以閱讀，而且充滿了不必要的抽象。你會看到：簡單的動作被包裝成命令模式、只要一行條件判斷就能解決的卡牌效果被寫成策略模式、可以直接建立的物件卻要透過工廠模式、還有一層又一層的繼承體系，讓追蹤資料流向變得幾乎不可能。原本只需要幾百行乾淨 Python 程式碼就能完成的事，最後膨脹成數千行、散落在幾十個檔案中——全都是因為每個問題到了他們手中，都變成了 Java OOP 鐵鎚下的釘子。

這個專案完美展示了把 Python 當成 Java 來寫，會產出多麼臃腫又難維護的學術垃圾。它是一座活生生的反模式博物館。

## 這堆混亂是誰寫的？

**這套軟體架構不是我設計的。**

大部分的程式碼是由我兩位隊友在過去的一個 Assignment（或 Sprint）中主導完成的。他們是很優秀的 Java 開發者——大學的 OOP 課程把物件導向程式設計教得非常好。好過頭了，也許。

不幸的是，他們把每一個 Java OOP 的習慣原封不動地搬到 Python。每個動作都需要一個類別。每種變化都需要一個抽象介面。每個集合都需要自己的子類別。而且他們不斷「添磚加瓦」——更多模式、更多抽象、更多檔案——最後讓整個專案變成一個難以閱讀的怪物。**我眼睜睜看著一個簡單的卡牌遊戲長出了觸手。**

## 這個儲存庫的目的

這個儲存庫存在的唯一理由是：**告訴你不該怎麼做。**

- **不要**把 Python 當成 Java 來寫。

- **不要**為每一件事都建立類別。

- **不要**不假思索地套用設計模式——先問自己：真的需要嗎？

- **不要**為了架構的純粹性而犧牲可讀性。

如果你是一個正在學習物件導向程式（OOP）設計的學生，這就是一個絕佳的**反面教材**，告訴你學到的東西**不該**怎麼用。研讀這些程式碼，學會辨識那些警示訊號，然後避免犯下同樣的錯誤。

## 最後的話

看看這個目錄結構。數一數有幾個檔案。問問自己：一個《Ticket to Ride》的克隆版本，真的需要一個 `actions` 資料夾、一個 `effects` 資料夾、一個 `comps` 資料夾、一個 `deck` 資料夾、一個 `utils` 資料夾、還有一個檔案重複出現的 `ui` 資料夾嗎？抽一張牌真的需要一個命令模式類別嗎？一張卡牌的效果真的需要策略模式嗎？

答案當然是：**不需要**。

而這就是這個儲存庫存在的意義。
