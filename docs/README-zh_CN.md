# overengineered-train-game

## 一个将 Java 设计模式硬搬到 Python 的警示案例

这个仓库包含的是海外知名桌游《Ticket to Ride》的 Python 实现。但我把它放在这里的目的**并非是**给出一个可运行的解决方案（实际上这个项目真的可以运行），而是提供一个严肃的**教育警示**：告诉你把 Java 的设计模式“生搬硬套”到 Python 上，结果会有多令人小脑萎缩（至少看得我力竭了）。

整个项目的代码存在多到逆天的过度设计，如同天书一样难以理解，而且有一大堆多余的抽象类。你会看到（包括但不限于）：简单的动作被包装成命令模式、只要一行条件判断就能解决的卡牌效果被写成策略模式、可以直接创建的对象却要通过工厂模式，还有一层又一层的继承体系（这不是 C++，事实上现代 C++ 也不推荐这么做），让调试和追踪数据流向变得非常困难。原本只需要几百行干净 Python 代码（那是我一开始写的）就能完成的事，最后膨胀成数千行、散落在几十个文件中——全都是因为每个问题到了他们手中，就非要使用 Java OOP 的设计原理来完成。所以说，**死读书是百害而无一利的**。

那么，我相信大家已经不难得出结论了：这个项目完美展示了把 Python 当成 Java 来写，会产出多么臃肿又难维护的学术垃圾。它是一座活生生的反模式博物馆。

## 这堆混乱是谁写的？

**这个巨 tm 抽象且逆天的架构真的不是我做出来的。**

大部分代码是由我的两位队友在过去的一个 Assignment（或者说，Sprint）中主导完成的。他们是很优秀的 Java 开发者——大学里的 OOP 课程把面向对象程序设计教得非常好。但是大家都知道的，《道德经》里面讲“物极必反”，过好反而是一件坏事。

于是，噩梦开始了：他们把每一个 Java OOP 的习惯原封不动地搬到 Python。每个动作都需要一个类。每种变化都需要一个抽象接口。每个集合都需要自己的子类。而且他们不断“添砖加瓦”——更多模式、更多抽象、更多文件——最后整个项目变成了一个“四不像”的怪胎。**我眼睁睁看着一个简单的卡牌游戏变得面目全非，却无能为力。**

## 这个仓库的目的

这个仓库存在的唯一理由是：**告诉你不该怎么做。**

- **不要**把 Python 当成 Java 来写。

- **不要**为每一样东西都建立类。

- **不要**无脑套用其他语言的设计模式——使用之前请三思：我真的需要这种设计模式吗？

- **不要**为了架构的纯粹性而牺牲可读性（不要让你的代码无病呻吟）。

如果你目前正在学习面向对象程序设计（C++ 也好，Python 也罢，但如果是 Java 你现在就可以划走了），那么这个仓库的代码就是一个绝佳的反面教材，告诉你学到的东西**不该**怎么用。研读这些代码，学会辨识那些警示信号，然后避免犯下同样的错误。

## 仓库政策

我之前提到过，该项目仅供教育与警示意义。因此，**请不要提交任何关于代码的 issues 或 pull requests**。但是，用来进行个人学习或研究是完全没有问题的。

**请不要去尝试修复代码里面的问题，然后来烦我**（算我求你了）——否则就违背了我建立这个仓库的初衷。整个项目在架构设计上就存在严重问题，旨在起到警示作用。如果你想知道正确的实现方式，请**去别的地方找**，或者**把这个仓库丢给 AI（ChatGPT, Claude, Gemini, Mistral AI, DeepSeek, GLM, MiniMax 等等），让它帮你去解决**。如果你想争论设计选择的好坏……那么恭喜你，你已经学到这个教训了。

## 最后的话

请大家仔细看看整个项目的目录结构，数一数有几个文件，同时问问自己：一个《Ticket to Ride》的克隆版本，真的需要一个 `actions` 文件夹、一个 `effects` 文件夹、一个 `comps` 文件夹、一个 `deck` 文件夹、一个 `utils` 文件夹、还有一个文件重复出现的 `ui` 文件夹吗？抽一张牌真的需要一个命令模式类吗？一张卡牌的效果真的需要策略模式吗？

答案当然是：**不需要**。

而这，就是这个仓库存在的意义。
